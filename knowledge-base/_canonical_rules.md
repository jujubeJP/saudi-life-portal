# サウジナビ KB 正本ルール（Canonical Record Rules）

> 同じ事実が複数のKBファイルに記載された場合の優先順位と解決ルールを定義する。

---

## 1. 基本原則

KBは **Single Source of Truth（唯一の真実の情報源）** を目指す。
同一の事実は原則として1つのファイルにのみ記載し、他のファイルからは参照する。

---

## 2. 優先順位（上が最優先）

| 優先度 | source_type | 例 | 理由 |
|--------|------------|-----|------|
| 1 | 公的一次情報 | 政府公報、法令、大使館公式通知 | 法的根拠がある |
| 2 | 公的二次情報 | 外務省まとめ、MOFA安全情報 | 一次情報を要約したもの |
| 3 | 準公的情報 | 商工会議所、JETRO、日本人会 | 信頼性が高い非政府情報 |
| 4 | メディア報道 | 英語/アラビア語ニュース | 速報性はあるが変動する |
| 5 | コミュニティ情報 | 在住者の体験、口コミ | 実用的だが主観を含む |
| 6 | AI生成・推論 | Claude等による要約・翻訳 | 元ソースの確認が必要 |

---

## 3. 重複検出ルール

### 3.1 同一事実の定義
以下のいずれかに該当する場合、「同一事実」とみなす:
- 同じ `source_url` を参照している
- `tags` の80%以上が重複し、`title` が類似（意味的に同じ）
- AI一貫性チェック（`ai_multicheck.py`）で `duplicates` として検出

### 3.2 重複発見時の対応

```
重複発見
  ├─ source_type の優先度が異なる → 高い方を正本（canonical）とする
  ├─ source_type が同じ
  │   ├─ checked_at が新しい方を正本とする
  │   └─ checked_at も同じ → confidence が高い方を正本とする
  └─ 内容が補完的（矛盾なし）→ 統合を検討
```

### 3.3 正本マーキング

正本ファイルの frontmatter に以下を追記:
```yaml
canonical: true
```

非正本（参照元）ファイルには:
```yaml
canonical: false
derived_from: "正本ファイルのid"
```

## 4. カテゴリ横断ルール

### 4.1 所有権の原則
各事実は **最も関連性の高いカテゴリ** が所有する:

| 事実の例 | 所有カテゴリ | 参照してよいカテゴリ |
|---------|------------|-------------------|
| eVisaの申請手順 | visa | travel-info（リンクのみ） |
| 緊急連絡先一覧 | safety | medical, living（抜粋OK） |
| 銀行口座開設 | finance | living（概要のみ） |
| SIMカード購入 | telecom | living（概要のみ） |
| 医療機関の場所 | medical | city-guide（リンクのみ） |

### 4.2 参照の書き方
非所有カテゴリから情報を参照する場合、本文中に以下の形式で記載:

```markdown
> 📌 詳細は [visa/evisa-requirements](../visa/evisa-requirements_YYYYMMDD.md) を参照
```

## 5. 矛盾解決プロセス

### 5.1 自動検出
`ai_multicheck.py --check consistency` が週次で矛盾を検出し、GitHub Issue を作成する。

### 5.2 手動解決フロー

```
1. Issue のチェックリストを確認
2. 矛盾する2つのファイルを開く
3. source_type の優先順位を比較
4. 正本を決定し、canonical フラグを設定
5. 非正本を更新（derived_from を追加、重複内容を削除）
6. Issue をクローズ
```

### 5.3 判断が難しい場合
- `notes_for_agent` に「要人間判断: [理由]」と記載
- GitHub Issue にラベル `needs-human-review` を追加
- YOSHIさんが最終判断

## 6. バージョン管理

### 6.1 ファイル名規則
```
{topic}_{YYYYMMDD}.md
```
- 同じトピックの新版を作る場合、新しい日付でファイルを作成
- 旧版は `_archive/` ディレクトリに移動

### 6.2 アーカイブルール
- `review_by` を過ぎて更新された場合、旧版はアーカイブ
- アーカイブファイルには `archived: true` を追加
- アーカイブから6ヶ月経過したファイルは削除可能

---

## 7. 品質基準

### 7.1 公開基準（confidence レベル）

| confidence | 意味 | 公開可否 |
|-----------|------|---------|
| high | 公的情報源で確認済み | ✅ 即公開 |
| medium | 信頼できるが二次情報 | ✅ 公開（注記付き） |
| low | 未確認・AI生成 | ⚠️ レビュー後に公開 |

### 7.2 公開前チェックリスト
- [ ] source_url にアクセスして内容が一致するか確認
- [ ] checked_at が3ヶ月以内か
- [ ] 矛盾する他のKBファイルがないか（ai_multicheck で確認）
- [ ] tags が適切に設定されているか
